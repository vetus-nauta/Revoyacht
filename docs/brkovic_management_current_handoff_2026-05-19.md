## 1. Самое важное для нового чата

Работать небольшими спринтами. Пользователь прямо просил делить работу на короткие шаги, чтобы не слететь по контексту.

Не трогать hero-блок с фото на странице менеджмента без отдельной просьбы. По hero уже было много итераций; текущая задача в основном про разметку, логику услуг, связь админки с публичной страницей, цены, профактуру и договор.

Админку менеджмента делать на русском языке. Публичная страница двуязычная: русский и английский. Английский перевод нужен адаптированный для англоязычной среды, не буквальный.

Критичная логика пользователя:

- База месяца — это начальный минимум.
- База месяца на публичной странице и в договоре должна быть одной строкой.
- Внутри базы: контроль яхты у причала, технический контроль, представление судовладельца.
- База зависит от длины, типа яхты и участия экипажа.
- Дополнительные месячные услуги идут поверх базы отдельными галочками.
- Разовые услуги идут отдельно, с количеством, и тоже считаются через коэффициенты типа и длины.
- Настройки админки задают только ценообразование и активность услуг; публичная страница и договор должны читать те же сохраненные данные.
- Цены дополнительных месячных и разовых услуг по умолчанию должны быть `0`, пользователь заполнит их сам.
- Коэффициенты типа, длины и экипажа сохранять рабочими.

## 4. Текущая формула публичного калькулятора

Файл:

```text
brkovic-stage/js/management.js
```

Состояние:

```js
state = {
  type: 'motor',
  length: '15-20',
  crew: 'none'
}
```

Ключи базы:

```js
CORE_MONTHLY_KEYS = ['boatWatch', 'technicalWatch', 'ownerRep']
```

Ключи дополнительных месячных:

```js
MONTHLY_ADDON_KEYS = [
  'batteryPower',
  'machineryStart',
  'ventilationHumidity',
  'sailingRigSupervision'
]
```

Ключи разовых услуг:

```js
ONE_TIME_KEYS = [
  'guestArrival',
  'deepInteriorCleaning',
  'exteriorDetailing',
  'seasonLaunch',
  'winterization',
  'emergencyVisit',
  'yardPeriod',
  'safetyReview',
  'tenderToys',
  'provisioningConcierge'
]
```

База месяца считается так:

```text
baseMonthly = round(length.base * type.factor * baseCrewFactor())
```

Пояснение: `length.base` уже является базовой ценой для диапазона длины. Это не сумма `boatWatch + technicalWatch + ownerRep`. Три базовые услуги используются для расчета влияния экипажа через `baseCrewFactor()`.

Дополнительные месячные:

```text
monthlyAddOn = round(service.price * length.factor * type.factor * serviceCrewFactor(service))
```

Разовые:

```text
oneTime = round(service.price * length.factor * type.factor) * quantity
```

Диапазон результата:

```json
"range": {
  "monthlyLow": 0.86,
  "monthlyHigh": 1.18,
  "oneTimeLow": 0.85,
  "oneTimeHigh": 1.2
}
```

Пользователь раньше спрашивал про непонятные цифры диапазона. Объяснение: диапазон — это нижняя/верхняя граница ориентировочного бюджета, а не годовая сумма и не умножение на 12. На публичной странице это месячный диапазон.

## 6. Админка менеджмента

Файлы:

```text
brkovic-stage/admin-management.html
brkovic-stage/js/admin-management.js
brkovic-stage/css/admin-management.css
brkovic-stage/management-admin-api.php
```

Короткая ссылка:

```text

```

Она делает redirect на:

```text
admin-management.html
```

Админка логинится тем же сценарием, что посты и комментарии:

```js
API_BASE = local ? '/admin-api-proxy.php?path=' : '/api'
login -> /auth/login
session check -> /auth/me
```

Для локального режима:

```js
LOCAL_HOSTS = ['brkovic-local.local', '127.0.0.1', 'localhost']
```

Локально админка должна сразу считаться авторизованной и работать с:

```text
/management-admin-api.php
```

Серверный API:

```text
brkovic-stage/management-admin-api.php
```

Методы:

- `GET` читает `data/management-pricing.json`;
- `PUT`/`POST` сохраняет JSON;
- сохраняет через временный файл и `rename`;
- валидирует группы и поля.

Если на FTP снова будет ошибка `Не удалось сохранить файл цен`, проверять права:

```text
brkovic-stage/data/
brkovic-stage/data/management-pricing.json
```

Нужны права на запись для PHP-процесса. API уже умеет прямую запись, временный файл и замену через `rename`; если директория не writable, он вернет ошибку.

## 7. Как устроена админка сейчас

Главная карточка админки:

- `Тип яхты`;
- `Длина`;
- `Доп. месячные`;
- `Итог месяца`;
- `Конструктор договора`.

Пользователь просил “не захламлять пространство”, поэтому настройки типа, длины, допов и разовых вынесены в всплывающие карточки.

Попапы:

- `typePricingPopup`;
- `lengthPricingPopup`;
- `monthlyAddOnPricingPopup`;
- `oneTimePricingPopup`.

Попапы:

- фиксированные поверх страницы;
- можно раздвигать через `resize: both`;
- можно двигать за шапку;
- закрываются кнопкой и `Escape`;
- имеют z-index management.

Конструктор договора скрывается/раскрывается кнопкой:

```text
Открыть договор / Скрыть договор
```

Пользователь просил договор переместить вниз страницы и сделать скрывающимся. Сейчас он внутри карточки `pricing-card--contract`, логика `moveContractToBottom()` пытается перемещать его после `.pricing-card--services`. Но в текущей HTML-разметке карточки `.pricing-card--services` может не быть. Это потенциально стоит проверить: договор и так находится последним в `admin-management-layout`, но функция может быть устаревшей.

## 16. Короткое резюме для старта нового чата

Если пользователь говорит “продолжаем страницу менеджмента”, начинай с проверки:

```text
brkovic-stage/services/yacht-management.html
brkovic-stage/js/management.js
brkovic-stage/admin-management.html
brkovic-stage/js/admin-management.js
brkovic-stage/management-admin-api.php
brkovic-stage/data/management-pricing.json
```

Главная модель:

```text
Тип яхты × длина × экипаж -> база месяца.
Доп. месячные -> цена услуги × тип × длина × экипаж.
Разовые -> цена услуги × тип × длина × количество.
Диапазон -> range low/high только для публичного ориентировочного результата.
Договор/профактура -> конкретные строки, нетто, дисконт, НДС, итог.
```

Главное не сломать:

- hero-фото;
- одну строку базы месяца;
- русскую админку;
- связь JSON цен -> публичная страница -> договор;
- раздельную логику базы, доп. месячных и разовых услуг.
